# Godot 全知识学习文档（系统学习版）

## 0. 说明
- “学习所有 Godot 知识”在实践中应拆成“完整知识体系 + 分阶段训练”。
- 本文档按 Godot 4 为主线，目标是让你从零到能独立完成可发布项目。
- 建议配套一个长期练习项目（例如：2D 动作、3D 解谜、平台跳跃）。

---

## 1. Godot 核心认知（先建立地图）
1. 引擎结构
- Scene Tree（场景树）：运行时所有节点组成的树。
- Node（节点）：最小功能单元，行为由脚本扩展。
- Scene（场景）：可复用的节点组合，可嵌套实例化。
- Resource（资源）：数据对象（纹理、音频、材质、脚本、动画等）。

2. 编程模型
- 生命周期：`_enter_tree` -> `_ready` -> `_process/_physics_process` -> `_exit_tree`。
- 信号机制：事件驱动，节点解耦通信。
- 数据驱动：通过 Resource/配置文件控制行为。

3. 你必须熟悉的关键词
- `Node2D`、`CharacterBody2D`、`RigidBody2D`、`Control`
- `PackedScene`、`AnimationPlayer`、`TileMap`、`NavigationAgent2D/3D`
- `ProjectSettings`、`InputMap`、`Autoload`

---

## 2. 编辑器与项目结构
1. 编辑器区域
- 场景树（Hierarchy）
- Inspector（属性）
- 文件系统（FileSystem）
- 脚本编辑器（Script）
- 输出/调试面板（Debugger）

2. 推荐目录结构
```text
res://
  scenes/
  scripts/
  assets/
    textures/
    audio/
    fonts/
  ui/
  data/
  shaders/
  addons/
```

3. 必做设置
- `Project -> Project Settings -> Input Map`：先定义操作键。
- 启用版本控制（Git），避免大改后无法回退。
- 定义统一命名规则（场景、脚本、资源）。

---

## 3. GDScript 完整能力清单
1. 基础语法
- 变量、常量、类型标注、数组、字典、循环、条件、函数。
- 缩进敏感：顶层不可乱缩进，函数体内部统一缩进风格。

2. 进阶语法
- `class_name`、静态函数、枚举、属性 `set/get`。
- `await` 异步流程（等待信号/定时器）。
- 匿名函数（Callable）、函数引用、工具脚本 `@tool`。

3. 面向对象
- 继承、组合、脚本复用。
- 将“可复用逻辑”封装成组件脚本。

4. 常见建议
- 游戏状态集中管理（如状态机）。
- 避免把大量逻辑塞进一个脚本。
- 明确“数据”和“行为”边界。

---

## 4. 场景系统与实例化
1. 场景拆分方法
- 主场景（关卡）+ 子场景（玩家、敌人、UI、特效）。
- 每个子场景只负责一类职责。

2. 动态实例化
- 使用 `PackedScene.instantiate()` 生成子弹/敌人/掉落物。
- 注意生命周期管理：生成、复用、销毁（对象池）。

3. 场景切换
- `change_scene_to_file()` 切换关卡/菜单。
- 搭配 Autoload 管理全局数据（存档、设置、成就）。

---

## 5. 2D 开发体系
1. 渲染与坐标
- `Node2D` 变换：位置、旋转、缩放。
- 摄像机：`Camera2D` 跟随、限制边界、平滑。

2. 物理与碰撞
- `CharacterBody2D`：角色控制（推荐玩家）。
- `RigidBody2D`：真实物理驱动。
- `Area2D`：检测触发区域（拾取、伤害圈）。
- `CollisionShape2D`：碰撞形状必须匹配精度需求。

3. 地图与关卡
- `TileMap`、地形、碰撞层。
- 分层地图（背景/前景/交互层）。

4. 2D 动画
- `AnimatedSprite2D`、`AnimationPlayer`。
- 过渡条件统一由状态机驱动（站立、跑、跳、受伤）。

---

## 6. 3D 开发体系
1. 常见节点
- `Node3D`、`CharacterBody3D`、`Camera3D`、`MeshInstance3D`。

2. 关键主题
- 3D 坐标系与朝向。
- 光照（Directional/Omni/Spot）与阴影成本。
- 材质与 PBR 基础（Albedo/Normal/Roughness/Metallic）。

3. 性能关键
- LOD、Occlusion、贴图压缩、网格合批。
- 3D 项目必须早做性能测试，不要等后期补救。

---

## 7. 输入系统
1. Input Map
- 为动作定义逻辑名：`move_left`、`jump`、`attack`。

2. 输入读取
- `Input.is_action_pressed()` 持续输入。
- `Input.is_action_just_pressed()` 单次触发。

3. 多端适配
- 键鼠、手柄、触屏分别抽象为同一动作层。

---

## 8. UI 系统（Control）
1. 布局核心
- 锚点（Anchors）、容器（Container）、最小尺寸规则。
- 避免硬编码像素位置，优先容器布局。

2. 常用节点
- `Label`、`Button`、`TextureRect`、`ProgressBar`、`Panel`。

3. UI 架构建议
- HUD、菜单、弹窗拆分不同场景。
- UI 与游戏逻辑通过信号交互，减少直接依赖。

---

## 9. 动画与时间线
1. `AnimationPlayer`
- 控制属性、方法调用、事件帧。

2. `Tween`
- 做轻量插值动画（位移、透明度、缩放）。

3. 动画树
- `AnimationTree` 适合复杂角色状态混合。

---

## 10. 音频系统
1. 节点
- `AudioStreamPlayer`、`AudioStreamPlayer2D/3D`。

2. 设计建议
- 区分 BGM、SFX、UI 声音总线。
- 通过 Audio Bus 做音量、滤镜、混响控制。

3. 基础规范
- 所有音量支持设置页调节并持久化存档。

---

## 11. 数据、存档与配置
1. 文件读写
- `FileAccess`、`DirAccess`。

2. 存档策略
- 存档数据对象化（玩家数据、关卡进度、设置项）。
- 加版本号，便于后续兼容升级。

3. 格式建议
- 小项目：JSON。
- 中大型：Resource + 二进制或结构化格式。

---

## 12. 信号与架构
1. 信号用途
- UI 按钮点击、受击事件、任务完成通知。

2. 解耦建议
- 不要所有节点互相直接访问。
- 用信号/事件总线/中介层组织依赖。

3. 状态机建议
- 玩家、敌人、Boss 行为统一状态机管理。

---

## 13. AI、导航与行为
1. 导航
- `NavigationRegion2D/3D` + `NavigationAgent2D/3D`。

2. AI 组织
- 感知（看到玩家）-> 决策（追逐/攻击）-> 执行（移动/技能）。

3. 常用方法
- 有限状态机（FSM）
- 行为树（BT，复杂 AI 推荐）

---

## 14. 网络与多人
1. 同步模型
- 权威服务器、P2P、主机-客户端。

2. Godot 高层多人 API
- RPC、同步变量、节点权限。

3. 注意事项
- 先做输入与状态同步最小闭环，再扩展玩法。

---

## 15. Shader 与渲染进阶
1. Shader 适用场景
- 溶解、描边、水波、扫描线、后处理效果。

2. 入门路径
- 先读懂材质参数，再写最简单顶点/片元逻辑。

3. 性能意识
- 特效越炫，越要测帧率和低端设备表现。

---

## 16. C#、GDExtension 与工具开发
1. C#
- 适合大型工程、团队规范、生态集成。

2. GDExtension（C++）
- 性能敏感模块可下沉到 C++。

3. EditorPlugin
- 可以做自定义编辑器工具，提升生产效率。

---

## 17. 性能优化与调试
1. 核心工具
- Profiler、Monitors、Frame Time。

2. 常见性能问题
- 频繁实例化销毁（改对象池）
- 过多 `_process` 节点（合并更新）
- 过度碰撞检测（减少检测范围和频率）

3. 调试习惯
- 每做完一个功能立刻压测，不要堆到最后。

---

## 18. 发布与跨平台
1. 导出流程
- 安装 Export Templates。
- 配置平台参数（Windows/Android/Web 等）。

2. 发布前检查清单
- 分辨率与 UI 自适应。
- 输入映射完整（键盘/手柄/触屏）。
- 资源缺失检查（贴图/音效/字体）。
- 存档读写权限与路径测试。

3. 版本管理
- 用语义化版本：`v0.1.0`、`v0.2.0`。
- 记录更新日志（新增、修复、破坏性变更）。

---

## 19. 常见错误速查
1. 脚本类型不匹配
- `Node2D` 脚本挂到 `Node` 会报错。

2. 代码缩进错误
- 顶层必须顶格；函数体内部缩进一致。

3. 背景遮挡游戏对象
- 用 `z_index` 或 `CanvasLayer` 调整层级。

4. 输入没响应
- 未在 `Input Map` 配置动作名。

5. 场景能运行但看不到对象
- 相机位置错误 / 节点不在视口 / 透明度为 0 / 图层遮挡。

---

## 20. 学习执行计划（建议）
1. 0-2 周（入门）
- 完成 3 个微型项目：接球、打砖块、平台跳跃基础移动。

2. 3-6 周（进阶）
- 完成 1 个完整 2D Demo：菜单、关卡、敌人、音效、存档。

3. 7-12 周（项目化）
- 引入状态机、对象池、数据驱动配置、性能优化。
- 完成可发布版本（Windows 或 Web）。

---

1. 初级

2. 中级
- 敌人巡逻+追击 AI、技能冷却、掉落系统、存档系统。

3. 高级

## 22. 你下一步最该做的事
1. 固定一个目标类型：先做 2D（推荐）。
2. 用你现有 `catch_game.gd` 做第一个“可完成”的小项目。
3. 每次只加一个功能，并写一条“变更记录”。

当你能独立完成“菜单 + 核心玩法 + 结算 + 存档 + 导出”，就已经具备实战开发能力。
